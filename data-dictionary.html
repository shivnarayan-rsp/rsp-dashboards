This page documents every KPI displayed across the RSP service dashboards — what it measures, why it matters, how it is calculated, and the exact SOQL query used to pull the data from Salesforce. Badges indicate whether a metric is a lag indicator (a result already achieved), a lead indicator (a behaviour that predicts future results), a snapshot (current state at time of refresh), or operational (a queue management metric).

Click any KPI row to expand it. All queries run against clearedgepacific.my.salesforce.com. The Work Type ID for Service Call is 08q7F0000000RDlQAM.
Service Dashboard — open cases snapshot
Total open cases
All non-closed cases across all territories and work types
Snapshot
Why it matters

The total queue size. The primary indicator of team workload and capacity strain. A rising trend week-on-week signals that cases are being created faster than they are being closed — a precursor to SLA breaches and customer dissatisfaction.

How it is calculated
COUNT(Case WHERE IsClosed = false)

Counts every Case record where the system field IsClosed is false. No work type filter — all case types are included.

SOQL query
SOQL
SELECT Service_Territory__c, COUNT(Id) CaseCount
FROM Case
WHERE IsClosed = false
GROUP BY Service_Territory__c
ORDER BY COUNT(Id) DESC
Salesforce fields used
ObjectFieldTypeNotes
CaseIsClosedBooleanSystem field. True when Status is a closed status value.
CaseService_Territory__cLookup → ServiceTerritoryCustom field. Lookup to ServiceTerritory object. Territory names resolved via separate query on ServiceTerritory.Name.
Open cases >72 hours
Open cases created more than 3 calendar days ago
Lead
Why it matters

A leading indicator of resolution SLA breaches and customer dissatisfaction. In Pacific Island MPS markets, a case that has been open for more than 3 days without resolution typically signals a parts availability problem, a travel constraint, or a case that has been forgotten. Monitoring this weekly creates accountability before the case becomes a complaint.

How it is calculated
COUNT(Case WHERE IsClosed = false AND CreatedDate < NOW − 72h)

The 72-hour threshold is calculated as a fixed datetime: the current timestamp minus 72 hours. Because SOQL does not support LAST_N_HOURS in aggregate queries, this is computed as a literal datetime at query time (e.g. 2026-03-20T00:00:00Z for a refresh on 23 March).

SOQL query
SOQL
-- Replace the datetime with: current refresh datetime minus 72 hours
SELECT Service_Territory__c, COUNT(Id) CaseCount
FROM Case
WHERE IsClosed = false
  AND CreatedDate < 2026-03-20T00:00:00Z
GROUP BY Service_Territory__c
ORDER BY COUNT(Id) DESC
Service Dashboard — service call performance
Average days to close — Service Call
Mean calendar days from case creation to closure, Service Call work type only
Lag
Why it matters

The primary efficiency lag measure for field service. It reflects the end-to-end time a customer waits from logging a fault to having it resolved. A lower number means faster resolution and higher customer satisfaction. Comparing month-on-month reveals whether operational improvements (parts stocking, scheduling, technician training) are having an effect.

Scoped to Service Call work type only to isolate reactive service visits from logistical tasks (toner delivery, asset installation) which have structurally different durations.

How it is calculated
For each closed case:
days = (ClosedDate − CreatedDate) in ms ÷ 86,400,000

Avg days to close = SUM(days) ÷ COUNT(cases)

Calculated in Apex (Anonymous Block) because SOQL cannot compute the difference between two datetime fields natively. Both CreatedDate and ClosedDate are queried as datetime values and the delta is computed in milliseconds then converted to days.

Apex query (used at refresh time)
Apex
List<Case> cases = [
  SELECT Id, CreatedDate, ClosedDate
  FROM Case
  WHERE IsClosed = true
    AND Work_Type__c = '08q7F0000000RDlQAM'  // Service Call
    AND ClosedDate >= 2026-03-01T00:00:00Z
    AND ClosedDate <  2026-04-01T00:00:00Z
];
Double total = 0;
for (Case c : cases) {
  total += (c.ClosedDate.getTime() - c.CreatedDate.getTime())
           / (1000.0 * 60 * 60 * 24);
}
Double avgDays = cases.size() > 0 ? total / cases.size() : 0;
Salesforce fields used
ObjectFieldTypeNotes
CaseCreatedDateDateTimeSystem field. Timestamp when the case was created.
CaseClosedDateDateTimeSystem field. Populated when case Status moves to a closed value.
CaseWork_Type__cLookup → WorkTypeCustom field. Filtered to Service Call ID 08q7F0000000RDlQAM.
CaseIsClosedBooleanSystem field.
First time fix rate (FTF)
% of closed Service Call cases resolved with exactly one Work Order
Lag
Why it matters

Industry benchmark for MPS service quality. A case with only one Work Order means the technician diagnosed and resolved the fault on a single visit. Multiple Work Orders indicate a return visit was required — meaning the customer's machine was down for longer, and technician travel costs were incurred twice.

Industry benchmark for managed print services is 75–80%. RSP is currently at 77.6% MTD, which is at the top of that range.

Current definition caveat: any second Work Order — including parts-only follow-ups or scheduled PMs — counts against FTF. Sridhar is reviewing whether non-callback second Work Orders should be excluded. This may increase the reported rate.

How it is calculated
FTF rate = cases with exactly 1 Work Order ÷ total closed Service Call cases

A case is a first-time fix if:
COUNT(WorkOrders WHERE CaseId = this case) = 1

Calculated in Apex. All closed Service Call cases for the period are queried, then their related WorkOrder records are counted via a second SOQL query. Cases with exactly one WorkOrder are counted as FTF.

Apex query (used at refresh time)
Apex
// Step 1: get closed Service Call cases for the period
List<Case> cases = [
  SELECT Id FROM Case
  WHERE IsClosed = true
    AND Work_Type__c = '08q7F0000000RDlQAM'
    AND ClosedDate >= 2026-03-01T00:00:00Z
];
// Step 2: count Work Orders per case
Map<Id, Integer> woCount = new Map<Id, Integer>();
for (Case c : cases) woCount.put(c.Id, 0);
for (WorkOrder wo : [
  SELECT CaseId FROM WorkOrder
  WHERE CaseId IN :woCount.keySet()
]) {
  woCount.put(wo.CaseId, woCount.get(wo.CaseId) + 1);
}
// Step 3: count cases with exactly 1 Work Order
Integer ftf = 0;
for (Integer n : woCount.values()) if (n == 1) ftf++;
Leaderboard — technician performance
Cases closed per technician
Total closed Service Call cases YTD, grouped by Assigned_Technician__c
Lag
Why it matters

The primary throughput measure per technician. Used to identify volume differences between technicians — which may reflect territory size, customer density, or scheduling efficiency. Not used in isolation: a high case count with a poor FTF rate or high callback rate is a worse outcome than a moderate case count with excellent quality metrics.

How it is calculated
COUNT(Case WHERE IsClosed = true AND Work_Type__c = 'Service Call' AND ClosedDate >= 2026-01-01) GROUP BY Assigned_Technician__c
Note on technician field

Technician name is stored in Assigned_Technician__c, a text field on the Case object — not a lookup to the User object. This means technician names are free-text strings, which can result in duplicates (e.g. "Ramiro Cataquis" and "Ramiro Cataquis (2)" appearing as separate entries). This is a data quality issue worth resolving in Salesforce by converting the field to a User lookup.

Callback rate
% of closed Service Call cases where Callback_ReadOnly__c = true
Lead
Why it matters

A callback occurs when a customer calls back after a case has been closed because the fault has reappeared or was not fully resolved. It is the strongest leading indicator of FTF rate — a high callback rate this month predicts a lower FTF rate next month. It also directly represents customer experience: every callback is a customer whose machine failed again.

Currently RSP's overall callback rate is 32.8% MTD (March 2026), which is high. A target of <15% is realistic for a well-run MPS operation.

How it is calculated
Callback rate = callbacks ÷ total closed Service Call cases

A case is a callback if:
Callback_ReadOnly__c = true
SOQL / Apex query
Apex
List<Case> cases = [
  SELECT Id, Assigned_Technician__c, Callback_ReadOnly__c,
         Service_Territory__c
  FROM Case
  WHERE IsClosed = true
    AND Work_Type__c = '08q7F0000000RDlQAM'
    AND ClosedDate >= 2026-01-01T00:00:00Z
    AND Assigned_Technician__c != null
];
// Count callbacks per technician
for (Case c : cases) {
  if (c.Callback_ReadOnly__c) callbacks++;
}
Salesforce fields used
ObjectFieldTypeNotes
CaseCallback_ReadOnly__cBooleanCustom field. Read-only formula or process-set field. True when a case has been flagged as a callback by the system or a user.
CaseAssigned_Technician__cText(40)Custom field. Free-text technician name. Not a User lookup — see data quality note above.
SLA violation rate
% of open Service Call cases with a response or resolution SLA breach
Lead
Why it matters

SLA violations on open cases are forward-looking risk indicators. A case with a resolution SLA breach that is still open is almost certainly going to generate a customer complaint and a poor CSAT score. Solomons and Tonga currently show 100% violation rates on open Service Calls — meaning every open service call in those territories has already breached at least one SLA milestone.

How it is calculated
Violation rate = cases with any SLA breach ÷ total open Service Call cases

A case has a violation if ANY of these are true:
SLA_Response_Violation__c = true
SLA_Resolution_Violation__c = true
SOQL query
SOQL
SELECT Id, Service_Territory__c,
       SLA_Response_Violation__c,
       SLA_Resolution_Violation__c
FROM Case
WHERE IsClosed = false
  AND Work_Type__c = '08q7F0000000RDlQAM'
Salesforce fields used
ObjectFieldTypeNotes
CaseSLA_Response_Violation__cBooleanCustom field. Set by Salesforce Entitlement process when the response milestone is breached.
CaseSLA_Processing_Violation__cBooleanCustom field. Processing milestone breach.
CaseSLA_Resolution_Violation__cBooleanCustom field. Resolution milestone breach — the most significant for customer impact.
Repeat visit rate by asset
Assets with 3+ closed Service Call cases in 2026 YTD
Lead
Why it matters

A machine generating 3+ service calls in a year is a repeat offender — it is consuming disproportionate technician time, likely frustrating the customer, and may indicate an asset at end-of-life that should be proactively replaced or refurbished. In a 4DX context this is a leading indicator of contract churn: customers with persistently faulty machines are the most likely to terminate or refuse contract renewal.

Currently 31 of 378 active assets (8.2%) have generated 3+ service calls YTD. These should be reviewed in the next customer business review.

How it is calculated
Group closed Service Call cases by AssetId.
Assets with COUNT >= 3 are flagged as repeat offenders.

Distribution buckets:
1 visit: single-visit assets
2 visits: moderate repeat
3+ visits: repeat offenders
SOQL / Apex query
Apex
List<Case> cases = [
  SELECT Id, AssetId, Asset.Name, Asset.SerialNumber,
         Service_Territory__c
  FROM Case
  WHERE IsClosed = true
    AND Work_Type__c = '08q7F0000000RDlQAM'
    AND ClosedDate >= 2026-01-01T00:00:00Z
    AND AssetId != null
];
// Group by asset and count
Map<Id, Integer> assetCount = new Map<Id, Integer>();
for (Case c : cases) {
  assetCount.put(c.AssetId,
    (assetCount.containsKey(c.AssetId) ? assetCount.get(c.AssetId) : 0) + 1
  );
}
// Filter for 3+ visits
for (Id aid : assetCount.keySet()) {
  if (assetCount.get(aid) >= 3) repeatOffenders.add(aid);
}
Open Cases — record-level detail
Open Cases list
Individual open case records with account, status, work type, age, and technician
Operational
Why it matters

Provides Sridhar with a drill-down view of every open case — filterable by territory, status, work type, and age. Each case number links directly to the Salesforce record for immediate action. This is the operational tool that the summary dashboard metrics point toward: when the dashboard shows 88% SLA violation rate, this list tells you which specific cases are breached.

Age calculation
Age = (query timestamp − CreatedDate) ÷ 3,600,000 milliseconds

Colour thresholds:
Green: < 3 days (72h)
Amber: 3–14 days
Red: > 14 days

Lead
Current status

The field Date_Time_Accepted__c exists on the Case object but is not being populated — 0 of 50 open Service Call cases have a value. A Salesforce validation rule or Flow should be implemented to require this field to be populated when a technician first accepts/acknowledges a case. Cases without a next step date
Open cases where Next_Step_Date__c is null — field not yet consistently used
Lead
Current status

The field Next_Step_Date__c exists on the Case object but is not consistently populated. In a 4DX model, every open case older than 24 hours should have a committed next step date — this creates explicit accountability. A case without a next step date is invisible on the team's radar.
