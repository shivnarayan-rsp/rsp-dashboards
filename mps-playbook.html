Daily Sridhar — Action Centre & Anomaly Check

Sridhar 5-minute morning check
  • 1
    Open MPS Actions. Review any new Open actions or Overdue items. Update status on actions you're progressing — click the status dropdown inline to save.
    → mps-actions.html
  • 2
    Check for meter_anomaly flags. Filter by Issue Type = "Meter Anomaly". If new flags exist, contact the site to arrange a meter read. Resolve the action once the read is obtained and note the value.
  • 3
    Note any high_visits flags from the overnight service log. If a machine has had 3+ visits this week, add a high_visits action and assign to yourself for investigation.

Weekly Sridhar + Avi — Machine P&L & Visit Flags

Sridhar + Avi Monday review — 15 minutes
  • 1
    Open MPS Machines. Filter by GM Band = "Critical (<60%)" and scan for any machines that have moved into the red band since last week.
    → mps-machine.html
  • 2
    Review high-visit machines. Sort by Visits (90d) descending. Any machine with >6 visits in 90 days is flagged. Log a high_visits action if one does not already exist. Assign to Sridhar.
  • 3
    Avi: check expiring contracts. Open MPS Contracts. Review the 61–90 day band. Any contract without a logged renewal action should get one today.
    → mps-contracts.html
  • 4
    Close out resolved actions. Any action where the work is done — mark as Resolved. Keep the actions list clean so Open count is a meaningful number.

Monthly SLT — Portfolio Scoreboard

SLT (Sridhar, Avi + GM) First Monday of the month — 30 minutes
  • 1
    Scoreboard check — Portfolio GM%. Open MPS Portfolio. Read the portfolio GM% KPI. Is it moving toward 80%? Note the direction vs last month.
    → mps-portfolio.html
  • 2
    Territory scorecards. Review each territory card. Note which territories are below 60% (critical) and which improved from last month. Identify the 1–2 highest-impact actions to drive this month.
  • 3
    GM band distribution. Check the count of machines in each band (>80% / 60–79% / <60%). The goal is to grow the ≥80% bucket every month.
  • 4
    Lead measure review. Review the three lead measures: (1) % of machines reviewed; (2) % renewals initiated ≥60d before expiry; (3) % of major accounts with QBR ≤90d. Are they moving?
  • 5
    Commitments. Each person states 1–2 specific commitments for the coming month to move the WIG. Log these as actions in mps-actions.html so they're tracked.
  • 6
    Rebuild dashboard data after the meeting to reflect the new month's starting position. See How to Rebuild below.

Renewal Avi — 90-Day Rolling Pipeline

Avi Every Monday — 10 minutes
  • 1
    Open MPS Contracts → Renewal Pipeline. Start with the 61–90 day band. These are the contracts that need a conversation started this week.
    → mps-contracts.html
  • 2
    For each contract in 61–90 days with no renewal action: Go to MPS Actions → Add Action. Set Issue Type = contract_renewal, fill the contract no and customer, assign to Avi, set due date 30 days before expiry.
  • 3
    Update notes on in-progress renewals. For contracts already in the Actions tracker, update the notes field with the latest conversation status (e.g. "Sent renewal quote 22 Mar, awaiting sign-off").
  • 4
    Escalate any contract entering ≤30 days without a renewal. This is a critical flag — raise it in the weekly team check-in immediately.

Renewal goal: no contract should reach expiry without a conversation logged at least 60 days beforehand. This is Lead Measure 2 for the WIG.

QBR Avi — Major Account Cadence

Avi Monthly scan — 10 minutes
  • 1
    Open MPS Contracts → QBR Tracker. Filter for Status = "Overdue" or "Due soon". These accounts need a QBR scheduled this month.
    → mps-contracts.html
  • 2
    Schedule the QBR meeting with the account contact. Standard QBR agenda: contract performance review · print volume trend · consumable usage · renewal outlook · any service issues · expansion discussion.
  • 3
    After the meeting, log the QBR date using the Edit button on the QBR row. Update Last QBR Date to today. Add notes summarising key discussion points and any commitments made.
  • 4
    If any account issues surface in the QBR (e.g. machine performance complaint, rate renegotiation request), log an account_review action in mps-actions.html immediately.

How-To Add or Resolve an Action

➕ Adding a new action
  1. Open MPS Actions in your browser.
  2. Click the + Add Action button (top right of the filter bar).
  3. Select the Issue Type — choose the closest match: Margin Critical, High Visits, Meter Anomaly, Contract Renewal, or Account Review.
  4. Fill in Assignee (Sridhar or Avi), Customer Name, and Territory — these are required.
  5. Add Serial No and/or Contract No if the action relates to a specific machine or contract.
  6. Write a clear Issue Detail and Suggested Action so the assignee knows what to do without needing to ask.
  7. Set a Due Date — default to 2 weeks out unless the situation is urgent.
  8. Click Save Action. The action appears in the table immediately.
If Supabase is not configured, actions are saved to the browser session only and will not persist on refresh. Contact Vijendra to set up the live connection.
✓ Resolving an action
  1. Find the action in the MPS Actions table (use filters or search).
  2. Click the Status dropdown in the Status column — change from Open or In Progress to Resolved.
  3. The change saves to Supabase instantly. The KPI counts update automatically.
  4. Optionally click View to open the detail panel and add a closing note describing what was done.

How-To Log a QBR Meeting

📋 Recording a completed QBR
  1. Open MPS Contracts and scroll to the QBR Cadence Tracker section.
  2. If the account is not yet in the tracker, click + Log Account and fill in the customer name, territory, and QBR cadence (quarterly or biannual).
  3. If the account is already listed, click Edit on that row.
  4. Update Last QBR Date to today's date.
  5. Add a brief note in the Notes field summarising the outcome (e.g. "Renewed 3yr MP contract, expanding to 2 more machines in July").
  6. Click Save. The QBR status badge will update from Overdue/Due soon to On track.
The QBR overdue threshold is 90 days for quarterly accounts and 180 days for biannual. The "Due soon" warning appears at 75% of the threshold (67 days / 135 days).

How-To Update the Labour Rate

The labour rate (NZD 49/hr) is embedded in the DATA object at the top of each dashboard HTML file. To update it:

⚙ Updating the labour rate in a dashboard file
  1. Open the relevant .html file in a text editor (e.g. VS Code, Notepad++).
  2. Use Find (Ctrl+F) to search for labourRateNZD.
  3. Change the value — e.g. from labourRateNZD: 49 to labourRateNZD: 52.
  4. Save the file and refresh the browser. All GM calculations update instantly.
  5. Repeat for each dashboard file that uses the labour rate (mps-portfolio.html, mps-machine.html).
If the dashboards are rebuilt from MCP at a later date, tell Claude the new rate before rebuilding so it's embedded correctly in the new files.

How-To Rebuild Dashboard Data

Dashboard data is static — embedded at the time the files were generated. To refresh with current BMS and Salesforce data:

🔄 Rebuilding with Claude Code
  1. Open a new Claude Code session in the MPS Dashboard project folder.
  2. Tell Claude: "Rebuild mps-portfolio.html with fresh data from BMS and Salesforce. Use the same design and DATA structure as the current file."
  3. Claude will query BMS (MySQL MCP) and Salesforce (SF MCP) and regenerate the file with today's figures. Check the as-of date in the header updates.
  4. Repeat for mps-machine.html and mps-contracts.html as needed.
  5. The Actions tracker and QBR section do not need rebuilding — they are live Supabase fetches.
Rebuild recommended: monthly (before the SLT review) or after a major contract change (new major account, large fleet change, rate renegotiation).

Reference GM Band Interpretation Guide

On Target — ≥ 80%
≥ 80%
No immediate action needed. Machine is contributing above the WIG threshold. Review annually at renewal. Look for expansion opportunities.
Watch — 60–79%
60–79%
Monitor monthly. Review visit frequency and consumable usage. Is the machine approaching end-of-life? Consider rate increase at next renewal. Do not let this persist for more than 3 months without a plan.
Critical — < 60%
< 60%
Act this week. Log a margin_critical action. Investigate: high visits? Low volume (under-utilised machine)? Consumable abuse? Old rate? Machine replacement candidate?

Common root causes by band

  • High visits (labour): Machine needs service upgrade or replacement. Escalate to vendor.
  • Low contract rate: Rate agreed years ago, no increase applied. Address at renewal — target CPI + 3% minimum.
  • High consumable use: Check if toner is being used for non-contracted sites. Verify delivery address.
  • Low volume: Machine is idle or underused — customer isn't printing. Engage before renewal to right-size or exit.
  • Multi-machine equal split: A high-volume machine in a shared contract appears unprofitable due to equal revenue split. Check meter reads before taking action.

AMP AMP Language Guide

The MPS review follows the AMP (Account Management Process) rhythm. Use this language in the monthly SLT meeting:

ElementWhat to coverDashboard location
Scoreboard Check Read the WIG metric: portfolio GM% vs 80% target. Up or down vs last month? By how much? MPS Portfolio → Portfolio GM% KPI card
Lead Measure Review (1) % machines with GM reviewed. (2) % renewals started ≥60d before expiry. (3) % major accounts with QBR ≤90d. MPS Machines count · MPS Contracts pipeline · QBR tracker
Commitments Each person states 1–2 specific actions they commit to this month that will move the WIG. Log in mps-actions.html. MPS Actions → Add Action
Account Rhythm Avi: which major accounts had QBR this month? Any expansion or churn risk surfaced? MPS Contracts → QBR Tracker
Renewal Rhythm Avi: how many contracts in the 61–90 day band? Are renewal conversations started? Any at risk? MPS Contracts → Renewal Pipeline (61–90d tab)

The meeting should take no longer than 30 minutes. The dashboard is the single source of truth — no spreadsheets, no separate reports. If a number is wrong, log it as an action and fix the source data.
